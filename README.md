## About

Advanced notes and examples on using FastAPI.


## Prerequisites

Code examples in this project have been developed and tested on Python >=3.12,<3.13. You may also need Docker for running additional services such as Redis, PostgreSQL, MongoDB etc.

**Note**: you may find some examples in this repository pointless and it's totally fine. The goal is to emphasize the integration and usage steps of different components in a FastAPI-based project.

## FastAPI Notes

* synchronous endpoints run in a separate thread, unlike asynchronous. This way you may need to adjust the max number of threads in the threadpool.

* If a third-party library (for interacting with a database, API, filesystem etc.) doesn't provide async interface, use regular synchronous endpoints.

* FastAPI uses [anyio](https://github.com/agronholm/anyio)-based threads to run blocking functions e.g. synchronous endpoints.

* FastAPI recommends to use asynchronous code wherever possible, as it runs directly in the *Event Loop* which runs in a single thread - main thread.

* *Error Handlers* and *Dependencies* should also be asynchronous even if they don't contain any asynchronous code. This makes them running in the same main thread. Otherwise they will be run in a separate thread. See [example](monitor-active-threads/).

* If a background task function is defined with `async def`, FastAPI will run it directly in the event loop, whereas if it is defined with normal `def`, FastAPI will use `run_in_threadpool()` and `await` the returned coroutine (same concept as API endpoints). 

* Another option when you need to run an `async def` function in the background, but not necessarily having it triggered after returning a FastAPI response (which is the case in BackgroundTasks), is to use `asyncio.create_task()`.

* It is important to limit the number of worker threads in the thread pools to the number of asynchronous tasks you wish to complete, based on the resources in your system, or on the number of resources you intend to use within your tasks.

* The number of worker threads in the `ThreadPoolExecutor` is not related to the number of CPUs or CPU cores in your system. You can configure the number of threads based on the number of tasks you need to execute.

* FastAPI recommends running blocking synchronous functions in the asynchronous context using Starlette's [run_in_threadpool()](https://github.com/encode/starlette/blob/b8ea367b4304a98653ec8ce9c794ad0ba6dcaf4b/starlette/concurrency.py#L35) function. Also [See](https://stackoverflow.com/a/77941425/17865804)

* Default number of worker threads of that external threadpool is 40.

* When running a FastAPI application via [uvicorn](https://github.com/Kludex/uvicorn), you can adjust the number of workers via `--workers` flag. The default value is the value of $WEB_CONCURRENCY environment variable or 1. 

*  When uploading files using `fastapi.UploadFile` class, the operations such as `await file.read()` and `await file.close()` under the hood utilize methods of the synchronous `fastapi.File` class in a separate thread, via `fastapi.concurrency.run_in_threadpool()`. Eventually the result is `await`ed into the main thread.

* FastAPI recommends uploading large files in chunks using Starlette's `request.stream()` method. This way the entire file will not be stored in memory. 

* *uvloop* is used to replace the event loop in the standard library `asyncio`. It is implemented with *Cython* and is very fast, which can increase the speed of asyncio by 2–4 times.

* Generate basic FastAPI project via **uvx**:
```bash
uvx fastapi-new awesomeapp
```
